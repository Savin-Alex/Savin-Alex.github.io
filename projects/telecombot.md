# DemoTel KZ · Telegram-бот поддержки

[RU](./telecombot.md) · [EN](../en/projects/telecombot.md)

Детерминированный (без LLM) Telegram-бот поддержки для мобильного оператора (демо-проект, 2026): безопасная маршрутизация обращений с приоритетом на риск и защиту персональных данных.

Идея — показать, что не каждую задачу нужно решать через GenAI. Для регулируемого телекома предсказуемость, тестируемость и контроль важнее «умных» ответов, поэтому ядро маршрутизации намеренно детерминированное.

## Паспорт проекта

| Поле | Содержание |
|---|---|
| **Проблема** | Поддержка телеком-оператора в мессенджере должна отвечать мгновенно, но часть обращений — высокорисковые (замена SIM, мошенничество, кража телефона). «Умный» LLM-бот здесь опасен: непредсказуем, его трудно протестировать и легко спровоцировать на выдачу того, чего нельзя. Плюс через чат проходят ПДн (номер, ИИН, OTP, карты). |
| **Цель** | Детерминированный бот, который безопасно закрывает рутину на L1 и гарантированно эскалирует высокорисковые сценарии в нужный канал, не пропуская ПДн в обработку и логи. |
| **В scope** | 50 сценариев (S01–S50) в 7 категориях + хендофф на оператора; маршрутизация по политике риска (L1 авто · Secure Web App · Fraud / L2); маскирование ПДн до обработки; secret-token проверка вебхука; набор тестов и P0-гейты. |
| **Вне scope** | Реальная интеграция с биллингом/CRM оператора; production-развёртывание; LLM-ответы; мультиязычная поддержка; голосовой канал. |

**Критерии успеха (со статусом):**
- Высокий риск никогда не закрывается автоматикой L1 → **выполнено** (bypass-правило + тесты)
- ПДн не попадают в обработку и логи → **выполнено** (маскирование MSISDN / ИИН / OTP / карт до роутинга)
- Поведение предсказуемо и воспроизводимо → **выполнено** (0 LLM в проде, 84/84 теста)
- Критичные сценарии закрыты до релиза → **выполнено** (5/5 P0-блокеров закрыто)
- Production-интеграция с системами оператора → **не выполнено** (демо-проект, вне scope)

## Моя роль

**Роль:** Junior IT PM · соло-проект (демо).

- Определил политику маршрутизации и классификацию риска по 50 сценариям.
- Принял ключевое продуктовое решение — без LLM в проде: детерминизм ради предсказуемости, тестируемости и контроля.
- Заложил privacy-by-design: маскирование ПДн до любой обработки, secret-token на вебхуке.
- Описал критерии приёмки и P0-гейты, собрал набор тестов на каждый сценарий и маршрут.

## Как устроено

Каждое обращение проходит детерминированный конвейер — без «догадок» модели:

- **Telegram Bot API** — приём апдейта через webhook.
- **FastAPI webhook** — проверка secret-token (запросы не от Telegram отбрасываются).
- **Маскирование ПДн** — MSISDN, ИИН, OTP и номера карт затираются до маршрутизации и логирования.
- **Политика и риск** — классификация обращения (LOW / HIGH) по сценарию.
- **Роутер сценариев (S01–S50)** — выбор готового ответа или эскалации.

**Маршруты исхода:**
- **L1 · авто** — self-service (баланс, пакеты, роуминг…).
- **Secure Web App** — операции с OTP / KYC / оплатой в защищённом мини-приложении.
- **Fraud · L2** — высокорисковые сценарии в защищённый канал с проверкой личности.

Ключевое правило: **высокий риск минует L1-автоматизацию** — бот не пытается «решить» SIM-swap или кражу сам, а сразу передаёт в Fraud / L2.

## Безопасность и контроль

- 0 LLM в проде — детерминированные ответы, которые можно полностью покрыть тестами.
- Маскирование ПДн до обработки; secret-token проверка вебхука.
- Высокорисковые сценарии по политике не закрываются на L1.
- 84/84 теста; 5/5 P0-блокеров закрыто перед релизом демо.

## Risk Register

| Риск | Митигация |
|---|---|
| Высокорисковый сценарий уходит в авто-ответ L1 | Политика bypass + тест на каждый HIGH-сценарий |
| Утечка ПДн в обработку или логи | Маскирование MSISDN / ИИН / OTP / карт до роутинга; PII-safe логирование |
| Поддельные запросы на вебхук | Проверка secret-token; отбрасывание не-Telegram трафика |
| Непредсказуемый ответ пользователю | Отказ от LLM в проде; детерминированный роутер на 50 сценариев |
| Регресс при изменении сценариев | 84/84 теста + P0-гейты как условие релиза |

## Ретро

- **Сработало:** заранее заданная политика риска и bypass-правило для HIGH-сценариев; маскирование ПДн на входе; полное тестовое покрытие сделало поведение воспроизводимым.
- **Улучшить дальше:** реальная интеграция с биллингом/CRM, мультиязычность (KZ/RU/EN), нагрузочное тестирование, production-развёртывание.
- **Ключевой вывод:** зрелость PM — это в том числе умение НЕ применять GenAI там, где важнее предсказуемость, безопасность и тестируемость.

## Статус

Демо/портфолио-прототип. Контур «обращение → вебхук → маскирование → риск → маршрут» работает и покрыт тестами; production-интеграция с системами оператора — вне scope.
