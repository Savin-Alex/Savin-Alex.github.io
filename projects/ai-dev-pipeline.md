# Multi-Agent AI Development Pipeline

MVP-проект по автоматизации небольшого delivery-цикла: от задачи в Jira до плана, генерации кода, ревью и Pull Request в GitHub.

## Бизнес-цель

Сократить lead time небольших инженерных задач за счёт автоматизации рутинного цикла plan → code → review → PR. Пайплайн берёт на себя шаблонные шаги, оставляя человеку контроль через review-цикл и явный fallback.

## Кратко о проекте

- Статус: MVP в работе.
- Цель: проверить, может ли узкий агентный SDLC-цикл надёжно обрабатывать небольшие задачи с понятными quality gates.
- Целевая цепочка: Jira task → planner → coder → reviewer → GitHub pull request.
- Стек: Python, Claude API, Jira Cloud, GitHub API, pytest.

## Проблема

Большинство AI coding demo выглядят как одношаговый сценарий:

- Вставить задачу в чат.
- Получить код.
- Вручную проверить, работает ли он.

Это полезно для личной продуктивности, но не является процессом поставки. В реальной работе есть intake, декомпозиция, coding, review, тестирование и работа с репозиторием.

Этот MVP отвечает на более практичный вопрос:

> Может ли узкая команда AI-агентов довести небольшую задачу от backlog до Pull Request, сохранив ревью, наблюдаемость и fallback на человека?

## Решение

Проект разбит на фазы — каждая даёт самостоятельный результат.

Scope первой версии:

- Один planner agent.
- Один coder agent.
- Один reviewer agent.
- Интеграция с Jira.
- Интеграция с GitHub.
- Один orchestrator loop.

В MVP нет параллельных swarm-сценариев, production-инфраструктуры и сложного deployment. Сначала нужно доказать, что работает контролируемый workflow.

## MVP Flow

- Задача берётся из Jira.
- Планировщик делает технический план.
- Кодер генерирует код и unit-тесты.
- Ревьюер проверяет результат.
- При approve создаётся Pull Request в GitHub.
- При ошибках задача возвращается в цикл доработки или уходит на ручное вмешательство.

## Архитектура

- `agents/` — промпты, схемы и обёртки для planner, coder и reviewer.
- `integrations/` — клиенты Jira и GitHub.
- `orchestrator/` — основной цикл пайплайна, retries, метрики и error handling.
- `tests/` — unit-тесты для клиентов и orchestration-логики.
- `docs/` — decision log, risk register и анализ.

Агентную логику, интеграции и orchestration можно тестировать отдельно.

## Фазы

### Прототип

Быстрый sanity check: Python-скрипт, который принимает описание задачи, отправляет его в Claude со structured output, запускает pytest в изолированном окружении и делает до трёх итераций с учётом ошибок тестов.

### Инфраструктура

Репозиторий, API-ключи, Jira-board, целевой GitHub-репозиторий, decision log, risk register, Kanban-доска.

### Интеграции

Тонкие клиенты Jira и GitHub без AI: чтение задач, перевод по статусам, комментарии, создание веток, коммиты, Pull Request, чтение статуса PR. Если связка Jira → GitHub не работает без AI, всему остальному нельзя доверять.

### Агенты

- `Planner` — превращает задачу в технический план и acceptance criteria.
- `Coder` — генерирует файлы реализации и unit-тесты.
- `Reviewer` — проверяет корректность, стиль, покрытие тестами и safety-аспекты.

Каждый агент возвращает structured output.

### Оркестратор

Полный цикл: задача из Jira → план → код → review-итерации → ветка и Pull Request → ссылка на PR обратно в Jira.

### Аналитика

Прогон системы на наборе небольших задач, сбор метрик, анализ провалов.

## Scope и trade-offs

MVP последовательный. Осознанные ограничения:

- Без параллельного исполнения.
- Без дополнительных агентов (DevOps, documentation).
- Без production-контейнеризации в первой версии.
- Без попытки автоматизировать большие и расплывчатые задачи.

## Надёжность и безопасность

AI-сгенерированный код рассматривается как недоверенный.

- Изолированный запуск тестов.
- Таймауты на тесты.
- Retry с backoff для API-ошибок.
- Жёсткий лимит на количество итераций.
- Fallback на человека, если цикл застрял.
- Secret scanning перед созданием Pull Request.

## Метрики

- Total task time.
- Количество review-итераций.
- Использование токенов.
- Стоимость в USD.
- Результат: success / failed / needs human.
- Количество сгенерированных файлов.
- Test coverage.

Главная бизнес-метрика — lead time savings: сравнение времени выполнения задачи вручную и через пайплайн.
